# Codex 执行计划 (ExecPlans) 规范

本文档描述了“执行计划”（ExecPlan）的要求。执行计划是一份设计文档，编码智能体（Coding Agent）可以根据该文档交付可运行的功能或系统变更。请将读者视为对本仓库完全陌生的新手：他们仅拥有当前的工作树和你提供的这一份 ExecPlan 文件。智能体没有对先前计划的记忆，也没有外部上下文。

## 如何使用 ExecPlan 和 PLANS.md

在编写可执行规范（ExecPlan）时，必须**逐字逐句**遵守 `PLANS.md`。如果该文件不在你的上下文中，请通过阅读整个 `PLANS.md` 文件来刷新记忆。在阅读（及重读）源代码时要彻底，以产出准确的规范。创建规范时，从骨架开始，并随着研究的深入充实内容。

在实施可执行规范（ExecPlan）时，不要向用户询问“下一步该做什么”；只需继续执行下一个里程碑。保持所有章节的内容为最新，在每个停顿点添加或拆分列表条目，以肯定地陈述已取得的进展和后续步骤。自主解决歧义，并频繁提交（Commit）。

在讨论可执行规范（ExecPlan）时，请在规范的日志中记录决策以供后人参考；对于规范的任何更改，其原因必须清晰明确。ExecPlan 是**生命文档**（Living Documents），应当始终能够**仅凭** ExecPlan 而无需任何其他工作即可重新开始。

在研究具有挑战性需求或重大未知因素的设计时，使用里程碑来实现原型验证（PoC）、“玩具级实现”等，以验证用户的提议是否可行。通过查找或获取库来阅读其源代码，进行深入研究，并包含原型以引导更完整的实现。

## 要求

**不可逾越的硬性要求：**

* **每份 ExecPlan 必须完全自洽（Self-contained）。** 自洽意味着它当前包含新手成功所需的所有知识和指令。
* **每份 ExecPlan 都是生命文档。** 贡献者必须随着工作的进展、发现的出现和设计决策的定稿而修订它。每次修订必须保持完全自洽。
* **每份 ExecPlan 必须使完全的新手能够在不具备本仓库预验知识的情况下，端到端地实现该功能。**
* **每份 ExecPlan 必须产生可证明的运行行为**，而不仅仅是为了“满足定义”而进行的逻辑更改。
* **每份 ExecPlan 必须使用平实的语言定义每个专业术语**，否则请勿使用。

**目的和意图第一。** 首先用几句话解释为什么这项工作对用户很重要：变更后用户可以做哪些以前做不到的事情，以及如何看到它发挥作用。然后引导读者完成实现该目标的准确步骤，包括要编辑什么、运行什么以及应该观察到什么。

执行计划的智能体可以列出文件、读取文件、搜索、运行项目和运行测试。它不知道任何先前的背景，也无法从早期的里程碑中推断你的意图。**重复你所依赖的任何假设。** 不要指向外部博客或文档；如果需要相关知识，请用你自己的话将其嵌入到计划中。如果 ExecPlan 构建在先前的 ExecPlan 之上且该文件已检入，请通过引用纳入。如果没有，你必须包含该计划中所有相关的上下文。

## 格式要求

格式和封装要求简单且严格。每份 ExecPlan 必须是一个标记为 `md` 的**单一围栏代码块**（以三个反引号开头和结尾）。**不要**在内部嵌套额外的三反引号代码围栏；当你需要显示命令、转录、差异（Diff）或代码时，请在单一段落内以**缩进块**的形式呈现。使用缩进来保持清晰，而不是使用内部代码围栏，以免提前关闭 ExecPlan 的代码块。在每个标题后使用两个换行符，使用 `#` 和 `##` 等标题级别，并使用正确的有序和无序列表语法。

当将 ExecPlan 写入 Markdown (.md) 文件，且该文件内容**仅包含**这一份 ExecPlan 时，应当**省略**三反引号。

使用平实的散文（Prose）写作。倾向于使用句子而非列表。除非简略会模糊含义，否则避免使用核对表（Checklist）、表格和长枚举。核对表仅允许出现在“进展”（Progress）章节中，且该章节必须使用核对表。叙述性章节必须坚持散文优先。

## 指南

**自洽性和平实语言至关重要。** 如果你引入了一个非普通英语短语（如 "daemon", "middleware", "RPC gateway", "filter graph"），请立即定义它，并提醒读者它在本仓库中是如何体现的（例如，命名出现它的文件或命令）。不要说“如前所述”或“根据架构文档”。即使是重复内容，也要在这里包含所需的解释。

避免常见的失败模式。不要依赖未定义的术语。不要把功能的描述写得过于狭隘，以至于代码虽然编译通过但没有任何实质意义。不要将关键决策外包给读者。当存在歧义时，在计划内部解决并解释原因。宁可过度解释用户可见的效果，也不要对偶然的实现细节规定不足。

**以可观察的结果作为计划的锚点。** 说明实现后用户可以做什么、运行的命令以及应看到的输出。验收标准应表述为人类可以验证的行为（“启动服务器后，导航至 http://localhost:8080/health 返回 HTTP 200 及正文 OK”），而不是内部属性（“添加了 HealthCheck 结构体”）。如果变更是内部的，解释如何演示其影响（例如，通过运行变更前失败、变更后通过的测试，并展示使用新行为的场景）。

明确指定仓库上下文。使用完整的仓库相对路径命名文件，精确命名函数和模块，并描述应在哪里创建新文件。如果涉及多个区域，请包含一小段方向性说明，解释这些部分是如何组合在一起的，以便新手能够自信地导航。运行命令时，显示工作目录和准确的命令行。当结果取决于环境时，陈述假设并提供合理的替代方案。

**保持幂等和安全。** 编写的步骤应能多次运行而不会造成损坏或偏移。如果某个步骤在执行一半时失败，请包含如何重试或调优的方法。如果需要迁移或破坏性操作，请详细说明备份或安全回退方案。优先选择保持测试通过的增量式变更。

验证不是可选的。包含运行测试、启动系统（如适用）以及观察其执行有用任务的指令。描述对任何新功能或能力的全面测试。包含预期的输出和错误消息，以便新手区分成功与失败。在可能的情况下，展示如何证明变更除了编译成功外确实有效（例如，通过一个小型的端到端场景、CLI 调用或 HTTP 请求/响应转录）。说明适用于项目工具链的准确测试命令以及如何解释结果。

**捕获证据。** 当你的步骤产生终端输出、短差异或日志时，将它们作为缩进示例包含在单一围栏块内。保持简明扼要，专注于证明成功的内容。如果你需要包含补丁（Patch），优先选择文件范围的差异或读者可以根据指令重建的小片段，而不是粘贴大的数据块。

## 里程碑

里程碑是叙述性的，而非官僚主义。如果你将工作分解为里程碑，请为每个里程碑引入一个简短的段落，描述其范围、里程碑结束时将存在的（而以前不存在的）内容、要运行的命令以及预期的验收结果。使其像故事一样可读：目标、工作、结果、证明。**“进展”（Progress）和“里程碑”是不同的：** 里程碑讲述故事，进展跟踪细粒度工作。两者必须同时存在。绝不要为了简洁而缩写里程碑，不要遗漏任何对未来实现至关重要的细节。

每个里程碑必须能够独立验证，并增量地实现执行计划的总体目标。

## 生命计划与设计决策

* ExecPlan 是生命文档。当你做出关键设计决策时，请更新计划以记录决策及其背后的思考。在 `决策日志`（Decision Log）章节中记录所有决策。
* ExecPlan 必须包含并维护 `进展`（Progress）章节、`惊喜与发现`（Surprises & Discoveries）章节、`决策日志`（Decision Log）以及 `成果与回顾`（Outcomes & Retrospective）章节。这些是强制性的。
* 当你发现优化器行为、性能权衡、意外错误或影响你方法的反向/取消语义时，请在 `惊喜与发现` 章节中记录这些观察结果，并附上简短的证据片段（测试输出是理想的选择）。
* 如果你在实施中途改变方向，请在 `决策日志` 中记录原因，并在 `进展` 中反映其影响。计划既是后续贡献者的指南，也是你的核对表。
* 在完成主要任务或完整计划时，编写一份 `成果与回顾` 条目，总结取得的成就、存在的差距和经验教训。

# 原型设计里程碑与并行实现

当能够降低较大变更的风险时，包含明确的原型设计里程碑是可以接受的（且通常是鼓励的）。例如：向依赖项添加低级运算符以验证可行性，或在测量优化器效果的同时探索两种组合顺序。保持原型是增量且可测试的。清晰地标注范围为“原型设计”；描述如何运行和观察结果；并陈述将原型晋升或放弃的标准。

优先选择增量代码变更，然后通过删减来保持测试通过。在大型迁移期间，并行实现（例如在迁移期间将适配器与旧路径保留在一起）是可以接受的，只要它们能降低风险或使测试能持续通过。描述如何验证这两条路径，以及如何通过测试安全地淘汰其中一条。当使用多个新库或功能领域时，考虑创建能够独立评估这些功能可行性的“探针”（Spikes），在隔离状态下证明外部库表现符合预期并实现了我们需要的功能。

## 优秀 ExecPlan 的骨架

```md
# <简短、动作导向的描述>

本 ExecPlan 是一份生命文档。随着工作的进行，`进展`、`惊喜与发现`、`决策日志` 和 `成果与回顾` 章节必须保持最新。

如果 PLANS.md 文件已检入仓库，请在此处引用从仓库根目录开始的路径，并注明本文件必须按照 PLANS.md 进行维护。

## 目的 / 大蓝图

用几句话解释用户在这次变更后能得到什么，以及他们如何看到它发挥作用。说明你将启用的用户可见行为。

## 进展

使用带复选框的列表来总结细粒度步骤。每个停顿点都必须记录在这里，即使需要将部分完成的任务拆分为两个（“已完成” vs “剩余”）。本节必须始终反映工作的实际当前状态。

- [x] (2025-10-01 13:00Z) 示例已完成步骤。
- [ ] 示例未完成步骤。
- [ ] 示例部分完成步骤（已完成：X；剩余：Y）。

使用时间戳来衡量进度率。

## 惊喜与发现

记录在实施过程中发现的意外行为、错误、优化或洞察。提供简明的证据。

- 观察：…
  证据：…

## 决策日志

按照以下格式记录工作期间做出的每个决策：

- 决策：…
  依据：…
  日期/作者：…

## 成果与回顾

在主要里程碑或完成时总结成果、差距和经验教训。将结果与原始目的进行对比。

## 上下文与方向

假设读者一无所知，描述与此任务相关的当前状态。通过仓库完整路径命名关键文件和模块。定义你将使用的任何非显而易见的术语。不要引用之前的计划。

## 工作计划

用散文形式描述编辑和添加的顺序。对于每次编辑，命名文件和位置（函数、模块）以及要插入或更改的内容。保持具体和精炼。

## 具体步骤

陈述要运行的准确命令以及运行地点（工作目录）。当命令生成输出时，显示一段简短的预期转录，以便读者进行比较。本节必须随着工作的进行而更新。

## 验证与验收

描述如何启动或运行系统以及观察什么。将验收表述为行为，并带有特定的输入和输出。如果涉及测试，说明“运行 <项目的测试命令> 并预期 <N> 个通过；新测试 <名称> 在变更前失败，变更后通过”。

## 幂等性与恢复

如果步骤可以安全重复，请说明。如果某个步骤具有风险，请提供安全的重试或回退路径。完成后保持环境整洁。

## 产出物与笔记

包含最重要的转录、差异或代码片段作为缩进示例。保持简明扼要，专注于证明成功的内容。

## 接口与依赖

保持指令性。命名要使用的库、模块和服务及其原因。指定里程碑结束时必须存在的类型、特质/接口和函数签名。优先使用稳定的名称和路径，如 `crate::module::function` 或 `package.submodule.Interface`。例如：

在 crates/foo/planner.rs 中，定义：

    pub trait Planner {
        fn plan(&self, observed: &Observed) -> Vec<Action>;
    }

```

如果你遵循上述指导，单一的无状态智能体——或人类新手——就可以从头到尾阅读你的 ExecPlan 并产出一个工作的、可观察的结果。这就是标准：**自洽、自足、引导新手、结果导向。**

当你修订计划时，必须确保你的更改全面反映在所有章节中（包括生命文档部分），并且必须在计划底部写一条注释，描述更改内容及其原因。执行计划不仅要描述“做了什么”，几乎每件事都要描述“为什么”。
